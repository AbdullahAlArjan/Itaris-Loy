## This session (Phase 2 — Identity & Access)

- **2026-07-18** — JWT is symmetric **HS256**, key from configuration (`Jwt:SigningKey`); dev uses a clearly-fake key in appsettings.Development.json, prod injects a real secret via env/secret store. Access token TTL **15 min** (doc 05), rotating refresh token TTL **30 days**. Three audiences: `customer`, `staff`, `admin`. _Why:_ single-service modular monolith — no need for asymmetric key distribution at this scale; revisit if a second service ever needs to validate tokens. _Source:_ doc 04 Part 1 (JWT + rotating refresh), doc 05 global conventions. **PLACEHOLDER:** 30-day refresh TTL is not stated in the frozen docs (doc 01 only says "customers must not be logged out often") — chosen value, flagged for confirmation.
- **2026-07-18** — Password/PIN hashing uses ASP.NET Identity's `PasswordHasher<T>` (PBKDF2-HMAC-SHA256), **not** the full ASP.NET Core Identity framework (no UserManager/SignInManager/Identity schema). _Why:_ our custom multi-audience `users` table (OTP customers + password owners + PIN staff) doesn't fit Identity's single-table UserManager model; doc 04's "ASP.NET Identity base" is read as "modeled on it + its password hasher." _Source:_ doc 04 Part 8 (users = ASP.NET Identity base), doc 01 Part 2 (staff email+password via ASP.NET Identity).
- **2026-07-18** — OTP codes: real 6-digit random, stored as SHA-256 hash (short-lived, low-entropy → full PBKDF2 unnecessary); a fixed dev-bypass code (`000000`) is honored **only** in the Development environment for pilot demos. _Why:_ doc 01 Part 2 explicitly asks for a dev/test bypass code; never in production. _Source:_ doc 01 Part 2.
- **2026-07-18** — OTP rate limiting: per-phone limit (3/hour) enforced by counting recent `otp_challenges` rows; per-IP limit (5/hour) via in-memory cache (`IMemoryCache`). _Why:_ doc 05 A1 limits (3/phone/h, 5/ip/h → `OTP_RATE_LIMITED`); IP counter is in-memory since it's a single-instance deployment at pilot scale (revisit for multi-instance). _Source:_ doc 05 A1.
- **2026-07-18** — Owner/staff login orchestration lives in the **Merchants** module (which depends on Identity), not Identity — even though doc 04 lists `OwnerLogin`/`StaffLogin` under Identity's commands. _Why:_ these flows join credential checks with merchant role/permission resolution; hosting them in Merchants keeps the root Identity module dependency-free. Identity exposes `ITokenIssuer` and `IUserDirectory` (public) so Merchants never touches identity's tables. _Source:_ doc 04 Part 6 (Identity is root, others depend on it), Part 7 (module boundaries).
- **2026-07-18** — `staff_invites` is placed in the **merchants** schema, not `identity` as doc 04 Part 8 lists it. _Why:_ an invite is defined by `merchant_id` + `staff_member_id` (merchant concepts); keeping it in Identity would force the root module to model merchant membership. Accepting an invite still provisions an identity user via `IUserDirectory`. _Source:_ deviation from doc 04 Part 8, flagged for review.
- **2026-07-18** — Refresh-token claims staleness: staff/owner permission changes take effect on next full login (bounded by the 15-min access token + refresh rotation), because the access-token claim set is snapshotted onto the refresh row. _Why:_ keeps refresh rotation a pure Identity operation (no cross-module call on every refresh). Acceptable at MVP scale; revisit if instant revocation is required. _Source:_ engineering decision.
- **2026-07-18** — Platform-admin auth: a seeded admin user (from `Admin:Email`/`Admin:Password` config) logs in via `/v1/auth/admin/login`, receiving an `admin`-audience token carrying `admin.merchants.create`, which gates the `/v1/admin` surface. _Why:_ doc 01 — merchant creation is admin-only (no self-serve); platform admins are internal Itaris staff. Dev credentials live in appsettings.Development.json; production injects real ones. _Source:_ doc 01 Part 2/3, doc 05 C10.
- **2026-07-21** — Subscription enforcement is manual in the MVP (doc 01: self-serve billing + payment processing are v2). A platform admin pauses/reactivates a merchant via `PATCH /v1/admin/merchants/{id}` (doc 05 C10), gated on `admin.merchants.manage`. This flips `merchants.status` between `active`/`paused` and is audited automatically. **Enforcement of what "paused" blocks** (new transactions/earning) lands with the Transactions module (Phase 4); **whether customers can still redeem existing points while paused is an open policy question** (leaning: allow redemption, block earning — not yet frozen). _Source:_ doc 01 Part 2, doc 05 C10.
- **2026-07-18** — Audit trail (doc 04 Ops): an EF `SaveChanges` interceptor captures every insert/update/delete of an entity done by a staff/admin actor and writes an `ops.audit_logs` row synchronously after the business save commits (before the response). Admin actions have `merchant_id = null` (doc 04: null=platform). Column names only in the payload — never secret values (hash/password/pin filtered). Best-effort: a failed audit write is logged, not propagated. _Why:_ doc 01 lists audit logs as an in-MVP fraud control. _Source:_ doc 04 Part 6/8, doc 01 Part 2. **Two implementation bugs found & fixed here:** (1) EF caches interceptor instances across scopes, so the interceptor must be a singleton reading the principal live from `IHttpContextAccessor` (a scoped `ICurrentUser` injected into it goes stale to the first — startup-seeding — scope, yielding a null actor forever). (2) `JwtBearer` defaults `MapInboundClaims=true`, which renamed `sub` and broke `CurrentUser.Id`; set `MapInboundClaims=false` so claim names round-trip.

## This session (Phase 3 — Loyalty foundation)

- **2026-07-21** — Loyalty programs: exactly one **active** program per merchant is enforced by a PostgreSQL **partial unique index** on `loyalty_programs (merchant_id) WHERE status = 'active'`, with a pre-check in the activate handler returning `PROGRAM_LIMIT` for a friendly error. _Why:_ doc 06 freeze (one active program) — belt-and-suspenders so the invariant can't be violated even by a race. _Source:_ doc 04 Part 6, doc 06 Part 16, doc 05 E3.
- **2026-07-21** — Rules are **immutable, versioned snapshots**: editing rules (doc 05 E2) inserts a new `loyalty_rules` row (version+1) and repoints `loyalty_programs.current_rule_id`; past rows are never mutated. _Why:_ transactions must keep the terms in force when they happened (fairness + audit); doc 04 calls the table "versioned rule snapshots". _Source:_ doc 04 Part 8.
- **2026-07-21** — The earn **calculator is a pure function** (`LoyaltyCalculator`, no I/O) over `RuleConfig`, using `decimal` (fils, never float). Points = `floor/round/ceil(amountMinor × pointsPerJod ÷ 1000)`; below `min_amount` earns nothing; stamps = `min(stampsPerVisit, maxStampsPerVisit)` per qualifying visit. DoD verified: 4.500 JOD at 1 pt/JOD floor → 4 points; stamps → 1. _Source:_ doc 04 Part 6 (CalculatePointsPreview pure), doc 06 Phase 3, doc 01 (points-per-JOD + rounding). **PLACEHOLDER:** the points-rate field name is clipped in doc 04 Part 8 — modeled as `pointsPerJod` per doc 01.
- **2026-07-21** — `customer_memberships.points_balance`/`stamps_filled` is a **cached projection**; the immutable source of truth is `points_ledger_entries`, which doc 04 places in the **transactions** schema (written by the Transactions module in Phase 4). So the Phase-3 "ledger append service + invariant checker" (doc 06) is deferred to Phase 4 where the ledger table lives — building it now would mean Loyalty writing another module's table. On join, the `welcome_bonus` is applied directly to the projection. _Source:_ doc 04 Part 6/8 (reconciling a doc 06↔doc 04 sequencing overlap).
- **2026-07-21** — JSON wire format: enums serialize as **strings** across the API (global `JsonStringEnumConverter`) to match doc 05's string-valued contract (e.g. rounding `"Floor"`). _Source:_ doc 05 global conventions.

## This session (Customers module — shadow / phone-only flow)

- **2026-07-21** — Phone-only customers (doc 01 "phone-number fallback, first-class"): a cashier enrolls by phone (doc 05 D3), which idempotently creates the identity **customer user keyed by phone** + a `customer_profiles` row with `is_shadow=true`. Because the user is keyed by phone, a later OTP login reuses the same user — so counter-earned memberships carry over with **no record merge**. The "claim-on-registration merge" (doc 06) reduces to flipping `is_shadow → false` + stamping `claimed_at` on first authenticated profile access. _Why:_ avoids a merge engine while satisfying the requirement; phone is the natural identity key. _Source:_ doc 04 Part 6 (Customers: shadow profiles + claim/merge), doc 01 Part 1, doc 05 D3/B1.
- **2026-07-21** — QR identity token (doc 05 B3): a short-lived (60 s) signed JWT (`aud=qr`, `sub=customerId`) using the same signing key, so the Phase-4 cashier resolve-qr flow can verify it. Rotation + short TTL is the fraud control (a screenshot is stale in seconds). _Source:_ doc 05 B3, doc 01 fraud controls.
- **2026-07-21** — Phone lookup (doc 05 D2) returns **masked** numbers (079•••4567: leading 3 + trailing 4). _Why:_ cashier can identify without seeing full PII. _Source:_ doc 05 D2, doc 01 (masked identifiers).
